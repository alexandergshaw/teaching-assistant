Most LLM calls route through `callLlm(req, provider)` in `src/lib/llm.ts` (single dispatcher; the old per-site Gemini `fetch` blocks were consolidated here on 2026-06-13). Provider is `"gemini"`, `"other"`, or `"embedded"`. `callLlm` ignores anything non-Gemini and falls back to Gemini, so non-grading features stay on Gemini regardless of the toggle.

The active provider is chosen via an in-app UI toggle (`src/app/components/ProviderToggle.tsx`), persisted in localStorage and read through `src/lib/llm-provider.ts` (`getStoredProvider` / `useLlmProvider`). The choice is passed to the server as an **argument**: server actions take a trailing `provider: LlmProvider = "gemini"` param; API routes (`ai-chat`, `parse-calendar`) read it from the request body/formData; the grading form sends it via a hidden `provider` input.

**The `"other"` provider is the Course Engine API** (`https://testing-knowledge-engine.vercel.app`), client in `src/lib/course-engine.ts` (env: `COURSE_ENGINE_URL`, optional `COURSE_ENGINE_API_KEY`). It is NOT wired into `callLlm` — `callLlm` ignores the provider and always uses Gemini (so any unmatched feature reaching it while the toggle is `"other"` transparently falls back to Gemini). The Course Engine covers only specific features, branched at the action level on `provider === "other"`:
- `generateCourseScheduleAction` → `POST /api/v1/schedule` (adapted to `CourseScheduleRow[]`; dates derived locally).
- lesson-planning deck (`page.tsx` `handleGenerateLesson` → `generateLectureDeckAction`) → `POST /api/v1/lecture`, downloads the `.pptx` directly (skips the editable preview + Gemini companion docs).
- lecture-planning package (`LecturePlanningTab` `handleGenerate` → `generateCourseMaterialsAction`) → `POST /api/v1/materials`, downloads `course-materials.zip`.
- `generateCourseRubricFromZipAction` → calls `/materials` and extracts `rubric.csv`.
- `generateCopilotProjectPromptAction` → `POST /api/v1/copilot-prompt` (deterministic, JSON in/out; returns `{ prompt }`).

Grading uses a **separate** dedicated service (NOT the Course Engine), client in `src/lib/grading-engine.ts` (env: `GRADING_ENGINE_URL` required, optional `GRADING_API_KEY`; error envelope `{error, messages}`). When the toggle is `"other"`, `gradeAction` (`src/app/actions.ts`) branches to `gradeViaGradingEngine` → `POST /api/v1/grade` (multipart: `submissions` zip + one of `rubric_csv`/`rubric_json`/`rubric_text`, auto-detected from an uploaded file or the pasted rubric). The deterministic per-criterion result is mapped to `GradingRun` via `gradingApiToRun` so the existing GradingTab matrix renders it (Files column shows "-", no full-credit checklist). The `GradingTab` shows a CSV/JSON rubric-file upload only when the toggle is `"other"`, and renders `state.warnings`.

**The `"embedded"` provider is the Embedded Deterministic Engine** (`src/lib/embedded-grader/`), an in-process grader with NO external service, key, or LLM. `gradeAction` branches on `provider === "embedded"` for both the zip and Canvas paths: it builds an `EmbeddedRubric` via `buildEmbeddedRubric` (a supplied rubric — Canvas/pasted/uploaded — wins; otherwise checks are generated from the assignment instructions by rule-based parsing), then `gradeEntriesEmbedded` runs each check (keyword/min_words/file_type/min_file_count/regex/code_symbol) and returns a `GradingRun` directly (keeping each student's userId + submittedFiles, so Canvas write-back and file previews work — better than "other"). Ingestion reuses `extractStudentEntries` (zip) / `extractCanvasEntries` (Canvas) added to `grade.ts` (deterministic, no filename-inference LLM call). When the rubric was generated from instructions it is surfaced via `state.generatedRubric` (rendered by `renderRubricText`); generation/mapping notes go to `state.warnings`. `buildEmbeddedRubric` caps the rubric at `MAX_CRITERIA` (4) checks for every source (generation orders most-concrete checks first), warning when extras are dropped. Grading shows a loader: the classic GradingTab flow uses the `pending` `loadingState` card (text varies by provider); the GitHub Repo source (`GithubGradingPanel.gradeAll`, which uses Gemini, not the embedded engine) has its own `busy === "grade"` loader card + button spinner.

Canvas **discussions** under the embedded provider take a separate path (`src/lib/embedded-grader/discussion.ts`): `gradeAction` branches on `detectCanvasUrlKind(canvasUrl) === "discussion"` and calls `buildDiscussionRubric` + `gradeDiscussion` instead of the generic file/keyword checks. Discussions are graded on 4 composite criteria (Participation, Timeliness, Engagement, Quality proxies) — each aggregating verifiable sub-signals (initial post, reply count, words, posted_by_due, distinct_days, replies_to_peers, mentions_peer, asks_question, includes_source, keyword coverage) with partial credit. This needs the structured thread data added to `fetchDiscussion`/`extractDiscussionActivity` in `canvas.ts` (`CanvasStudentWork.discussion`: initial posts vs replies, timestamps, parent author; plus the topic `dueAt`). The "Quality proxies" column is surface-only and always carries a warning that it is not a content-quality judgment.

As of 2026-07-01 the embedded provider covers **every** LLM feature (no silent Gemini fallback), via `src/lib/embedded/` (non-grading deterministic modules) alongside `src/lib/embedded-grader/` (grading). Each action branches on `provider === "embedded"`:
- Rubric generation: `generateRubric` in `grade.ts` routes to `generateEmbeddedRubricText` (reuses `buildRubricFromInstructions` + `renderRubricText`, capped at `MAX_CRITERIA`, now centralized in `embedded-grader/rubric.ts`). Covers assignment/course-project/zip/repo rubric + repo-grading fallbacks.
- Repo grading: `gradeReposAction`/`gradeRepoAction` use `gradeEntriesEmbedded`; `ingestRepo` now returns structured `files: RepoFile[]` so file-type/count checks + previews work (`repoDigestToEmbeddedEntry` in `actions.ts`).
- `embedded/meeting.ts` (meeting-request classifier), `content.ts` (module intro + assignment), `deck.ts` (lesson-plan slides + examples), `docs.ts` (markdown document / module-intro / assignment-sheet), `communication.ts` (announcement + message/meeting reply), `accessibility.ts` (alt text from filename, link text from URL), `course.ts` (fixed project rubric, repo outline, copilot prompt), `syllabus.ts` (label-based field detection), `schedule.ts` (computed weekly schedule), `calendar.ts` (rule-based dated-event extraction; wired into `calendar-parser.ts`).
- Generative/transformational features are now substantive under embedded (2026-07-01 sweep): chat answers extractively via `embedded/answer.ts` (`answerFromContext`: sentence retrieval weighted by term rarity, summary + define intents); revisions apply concrete edit commands via `embedded/revise.ts` (`applyTextRevision`/`applySlidesRevision`/`applyHtmlRevision`: replace/remove/add/retitle/shorten; unparseable instructions leave content unchanged); paragraph rewrite uses `copyedit`. Decks/examples mine real definitions (`extractDefinitions`) and fenced code blocks (`extractCodeBlocks`) from the source; docs carry curated real free-resource links + factual per-tech applications; overviews use the extractive `summarize`; phrasing varies deterministically via `pick` (FNV-1a-seeded). `generateEmbeddedRubricText` renders the LLM's weighted 3-tier format (round-trips through the rubric parser); calendar parses date ranges into `endDate`; grading comments vary per student around a fixed factual core. Still honest limits: image-alt from pixels returns a switch-provider error, `autoFixOfficeFileAction` skips vision alt but keeps deterministic title/heading fixes, and the engine never composes novel answers/solutions not present in the source.
- Course Engine ("other")-only actions with no `provider` param (`generateLectureDeckAction`, `generateCourseMaterialsAction`) are unreachable under embedded. Each `embedded/*` module has a co-located `*.test.ts`.
